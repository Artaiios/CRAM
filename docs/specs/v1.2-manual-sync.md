# V1.2 — Manueller Online-Sync mit Awareness

Stand: 12. Mai 2026. **Spec-Status: implementiert** (Branch `v1-2-manual-sync`, Phasen 1.1–3.4 abgeschlossen). Was geschippt vs. deferred:

- ✓ Sync-Source-Abstraktion + Multi-Source-CRUD
- ✓ S1 (HTTP) und S5 (File-System-Access-API) Backends
- ✓ Encryption (PBKDF2 + AES-GCM) Default ON
- ✓ Sync-Bundle Share/Import via JSON-Text und Datei
- ✓ Awareness-Indikator im Header
- ✓ Browser-Support-Banner (Firefox / Safari)
- **Deferred:** QR-Channel für Sync-Bundle (Magic-Link `cram://`) — die Bundle-Distribution per JSON-Text/Datei reicht für V1.2; QR wird ein separater Commit nach V1.2-Release, wenn die Praxis es einfordert.

## Ziel

Sync-Source-Architektur in CRAM einführen, um die in V1.0 vorhandenen manuellen Sync-Channels (Code, QR, JSON) durch eine HTTP- und File-System-basierte Source-Abstraktion zu ergänzen. **Keine Auto-Magie** — User klickt "Holen" und "Schicken" explizit, sieht Versionsstand und Fingerprint vor jeder Aktion. Vertrauen aufbauen, bevor V2.0 die gleichen Pfade automatisiert.

## Nicht-Ziele

- Auto-Polling, Auto-Push, Hintergrund-Sync — alles V2.0
- Konflikt-Auflösungs-Logik (Pull-before-push, If-Match) — V2.0
- Multi-Source — V2.0
- S2-Backend (S3-presigned-URLs) — V2.1
- OAuth-Integrationen (OneDrive/Drive direkt) — Anti-Roadmap

## User Stories

**US-1: Stabsleiter richtet Sync-Source einmalig ein.**
Bob (Stabsleiter) klickt im Settings-Modal auf "Sync-Sources verwalten", trägt URL seines internen `nginx`-Servers ein, vergibt eine Passphrase, klickt "Speichern und Test". CRAM macht einen Probe-PUT, meldet "Verbindung okay, verschlüsselter Stand geschrieben". Fertig.

**US-2: Stabsleiter teilt Setup mit Stab.**
Bob klickt "Sync-Bundle teilen". CRAM zeigt einen QR-Code. Alice öffnet CRAM auf ihrem Gerät, klickt "Sync-Bundle scannen", scannt den Code. CRAM importiert die Sync-Source-Konfiguration und prüft, dass der lokale Config-Fingerprint zur Source passt. Fertig — Alice ist im Sync-Setup.

**US-3: Stabsleiter pusht aktuelle Daten.**
Bob hat Personen-Daten geändert. Klickt im Sync-Modal auf "Stand auf Server schicken". CRAM pullt erst kurz, prüft Versionen, schreibt dann lokal v18 als verschlüsseltes JSON auf den Server. Awareness-Indikator springt auf "synced (v18, gerade jetzt)".

**US-4: Stabsmitglied holt aktuellen Stand.**
Alice öffnet CRAM, sieht im Header den Awareness-Indikator orange: "out-of-sync (Server: v18, lokal: v17, vor 2 h)". Klickt drauf oder öffnet das Sync-Modal, klickt "Stand vom Server holen". CRAM lädt v18, entschlüsselt, prüft Fingerprint, überschreibt lokalen Stand. Indikator wird grün.

**US-5: Awareness ohne Aktion.**
Alice öffnet CRAM während eines laufenden Incidents. CRAM macht **kein automatisches Polling**, aber führt beim **Tab-Visibility-Wechsel** einen leichtgewichtigen `HEAD`-Request gegen die Source aus, um die Server-Version zu lesen (kein Volltext-Pull). Indikator zeigt aktuell, ob Server-Stand neuer ist. Alice entscheidet selbst, ob sie pullt.

## Architektur

### Sync-Source-Abstraktion

Jede Sync-Source ist ein JS-Objekt mit fester Schnittstelle:

```javascript
{
  id: "cram-source-1234",                  // intern eindeutig
  type: "http" | "fileSystem",             // Backend-Typ
  label: "Hauptserver (nginx intern)",     // user-sichtbar
  config: { /* backend-spezifisch */ },
  encryption: {
    enabled: true,                         // Default ON
    passphrase: null                       // im RAM zur Laufzeit, nie persistiert
  },
  
  async read() {
    // Returns: { payload: object, version: int, timestamp: number, fingerprint: string }
    //          oder null wenn Source leer ist
    //          throws bei Netzwerkfehler / Auth-Fehler / Decrypt-Fehler
  },
  
  async write(envelope) {
    // envelope = { payload, version, timestamp, fingerprint }
    // Schreibt verschlüsselten Inhalt zur Source
    // throws bei Netzwerkfehler / Auth-Fehler / Encrypt-Fehler
  },
  
  async probe() {
    // Returns: { reachable: bool, latencyMs: int, serverVersion: int|null }
    // Leichtgewichtig — HEAD-Request, kein Body-Download
  }
}
```

### Storage-Format

Eine Sync-Source enthält ein **Envelope-JSON**:

```json
{
  "format": "cram-sync-v1",
  "version": 18,
  "timestamp": 1746710400000,
  "fingerprint": "a3f1",
  "encrypted": true,
  "payload": "<base64 AES-GCM ciphertext>",
  "iv": "<base64 96-bit nonce>",
  "salt": "<base64 16-byte salt für PBKDF2>"
}
```

Wenn `encrypted: false`: `payload` ist direkt das Klartext-CRAM-State-JSON.

### Versions-Mechanik

- **`version`** ist ein monoton steigender Integer pro Source.
- Lokale Mutation → Increment der lokalen Version → bei Push wird diese Version übertragen.
- Konflikt-Erkennung in V1.2 ist primitiv: vor Push wird einmal probiert, der Server-Version zu lesen. Wenn Server-Version > lokale Version, wird **vor dem Push** ein Pull-Dialog angeboten ("Server hat v18, du hast v17 — erst pullen?"). Kein If-Match, kein automatisches Merge.
- `timestamp` ist Tiebreaker (wenn Versionen identisch, was real nur durch Setup-Fehler passiert).
- **`fingerprint`** ist 4-Char-Hash der lokalen Config-Struktur. Muss zwischen Source und Lokal matchen, sonst wird Sync verweigert ("Du syncst gegen einen anderen Stab").

### Lokale Persistenz

Neuer localStorage-Key: **`cram.sync.v1`**.

Inhalt:
```json
{
  "sources": [
    {
      "id": "src-001",
      "type": "http",
      "label": "Hauptserver intern",
      "config": {
        "url": "https://nginx.corp.example.com/cram/state.json",
        "authType": "bearer",
        "authToken": "<token, encrypted at rest mit Browser-Profile-Schlüssel>"
      },
      "encryption": { "enabled": true }
    }
  ],
  "lastSyncedVersion": 17,
  "lastSyncedTimestamp": 1746704800000,
  "lastSyncedSourceId": "src-001"
}
```

**Hinweis zu Token-Storage:** localStorage in Klartext ist XSS-exponiert. Der CSP-Schutz minimiert das. Der Auth-Token wird zusätzlich mit einem aus dem Browser-Origin abgeleiteten Schlüssel verschlüsselt. **Keine perfekte Sicherheit, aber Schutz vor naivem File-Lese-Zugriff.** Das Risiko-Modell ist klar dokumentiert: wer DOM-Zugriff hat, kann den Token holen — Mitigation ist die CSP, nicht die localStorage-Kryptographie.

### Encryption-Details

- **Key Derivation:** PBKDF2-HMAC-SHA256, 250 000 Iterationen, 16-Byte-Salt (pro Sync-Source, nicht pro Schreibvorgang).
- **Cipher:** AES-256-GCM, 96-Bit-Nonce (pro Schreibvorgang neu).
- **API:** ausschließlich `crypto.subtle` (WebCrypto), keine externe Library.
- **Salt-Persistenz:** Salt wird im Envelope mitgeschrieben, beim Lesen verwendet. Erste Schreibung legt Salt fest.
- **Passphrase-Lebensdauer:** im RAM, beim Browser-Close weg. User muss bei Session-Start einmalig die Passphrase eingeben → Settings-Modal öffnet sich automatisch wenn Sync-Source vorhanden aber kein Key in der aktuellen Session.

## Backend S1: HTTP PUT/GET

**Konfiguration:**
- `url` (Pflicht): vollständige HTTPS-URL zur State-Datei (z.B. `https://nginx.corp.example.com/cram/state.json`)
- `authType`: `none` | `basic` | `bearer`
- `authUser`, `authPass` (für Basic) oder `authToken` (für Bearer)

**Operationen:**

| Operation | HTTP-Methode | Erwartete Server-Antwort |
|---|---|---|
| `read()` | `GET <url>` | 200 mit Envelope-JSON, oder 404 (Source noch leer) |
| `write(env)` | `PUT <url>` mit `Content-Type: application/json` | 200/201/204 |
| `probe()` | `HEAD <url>` | 200/404; aus Headern Last-Modified extrahieren |

**Server-Anforderungen:**
- Unterstützung für GET, HEAD, PUT (also kein reines Static-Hosting — nginx mit `dav_methods PUT;`, Caddy mit `file_server` reicht **nicht**, Caddy braucht `webdav`-Plugin)
- CORS-Header gesetzt:
  ```
  Access-Control-Allow-Origin: <CRAMs Origin oder *>
  Access-Control-Allow-Methods: GET, HEAD, PUT, OPTIONS
  Access-Control-Allow-Headers: Authorization, Content-Type, If-Match, If-None-Match
  Access-Control-Expose-Headers: ETag, Last-Modified
  ```
- Auth-Mechanismus optional, aber empfohlen (Basic über HTTPS oder Bearer)

**Beispiel-nginx-Konfiguration** (kommt in Doku):
```nginx
location /cram/ {
    dav_methods PUT DELETE;
    dav_access user:rw group:rw all:r;
    create_full_put_path on;
    auth_basic "CRAM";
    auth_basic_user_file /etc/nginx/.htpasswd-cram;
    add_header Access-Control-Allow-Origin "*" always;
    add_header Access-Control-Allow-Methods "GET, HEAD, PUT, OPTIONS" always;
    add_header Access-Control-Allow-Headers "Authorization, Content-Type" always;
}
```

## Backend S5: File System Access API

**Konfiguration:**
- `directoryHandle` (Pflicht): persistierter Handle (in IndexedDB)
- `fileName`: meist `cram-sync.json`

**Operationen:**

| Operation | Implementierung |
|---|---|
| `read()` | `directoryHandle.getFileHandle(fileName).getFile().text()` → JSON.parse → Envelope |
| `write(env)` | `directoryHandle.getFileHandle(fileName, {create:true}).createWritable().write(...)` |
| `probe()` | `getFile().lastModified` lesen |

**Browser-Verhalten:**
- **Chrome/Edge:** Permission persistiert automatisch zwischen Sessions
- **Safari:** Permission muss bei jedem Browser-Start neu erteilt werden — UI muss das auffangen ("Verzeichnis-Zugriff bestätigen")
- **Firefox:** API nicht unterstützt — beim Versuch S5 anzulegen wird Modal angezeigt: "Diese Funktion benötigt einen Chromium-basierten Browser oder Safari. Empfehlung: Edge, Chrome oder Safari."

**Setup-Flow:**
1. User klickt "Lokales Verzeichnis hinzufügen"
2. Browser-Native Directory-Picker öffnet sich
3. User wählt einen Ordner (typisch: ein OneDrive-/Dropbox-/Drive-Sync-Folder)
4. CRAM speichert Handle in IndexedDB
5. Probe schreibt eine `.cram-sync-test`-Datei und löscht sie wieder, um Schreibrecht zu verifizieren

## Sync-Bundle (Onboarding)

Ein Sync-Bundle ist ein kompaktes Onboarding-Artefakt. **Enthält Auth-Credentials → ist vertraulich.**

Schema:
```json
{
  "format": "cram-bundle-v1",
  "type": "http",
  "label": "Hauptserver intern",
  "config": {
    "url": "https://nginx.corp.example.com/cram/state.json",
    "authType": "bearer",
    "authToken": "<...>"
  },
  "encryption": {
    "enabled": true,
    "passphrase": "<...>"
  },
  "configFingerprint": "a3f1"
}
```

**Distributionswege:**

1. **QR-Code** — Bundle wird komprimiert (gzip via fflate, schon eingebaut), in Base64 codiert, als QR-Code dargestellt. Receiver scannt mit existierendem QR-Reader-Code-Pfad.
2. **Magic-Link** — `cram://join?bundle=<base64gz>`. Klick öffnet CRAM mit vorausgefüllter Import-Aufforderung. Funktioniert nur, wenn Custom-Protocol-Handler registriert ist (auf dem System einmalig). Alternativ: `https://[CRAM-Origin]/?join=<base64gz>` Query-Parameter.
3. **JSON-Datei** — Manueller Download/Upload, wie bisheriger JSON-Export.

**Receiver-Validierung:**
- Decompress + JSON-parse
- `format`-Check
- Probe gegen die Source: erreichbar, Fingerprint match
- Bei Fingerprint-Mismatch: deutliche Warnung "Du würdest gegen einen anderen Stab syncen — abbrechen empfohlen"

## UI-Änderungen

### Header

Neuer Awareness-Indikator **links** der bestehenden Status-Pills, durch vertikalen Trenner abgegrenzt:

| Zustand | Symbol | Tooltip |
|---|---|---|
| `synced` | grünes Häkchen-im-Kreis | "Synchron mit [Source-Label] (v18, vor 23s)" |
| `syncing` | gelber Spinner | "Sync läuft…" |
| `out-of-sync` | oranger Pfeil-im-Kreis | "Server hat neueren Stand (v18 vs lokal v17)" |
| `error` | roter Kreis-mit-X | "Sync-Fehler: [Fehlermeldung]" |
| `disabled` | grauer durchgestrichener Kreis | "Keine Sync-Source konfiguriert" |

**Indikator ist nicht klickbar** — reine Anzeige. Sync-Aktionen erfolgen explizit im Sync-Modal.

### Sync-Modal (vorhandenes Modal erweitern)

Vier Tabs (statt heute drei):
1. **Code** (bestehend)
2. **QR** (bestehend)
3. **Datei** (bestehend)
4. **Online** (neu)

Im "Online"-Tab:
- Liste aller konfigurierten Sources (label, type, status)
- Pro Source: zwei Buttons "Holen" / "Schicken", Versions-Anzeige (lokal vs. server)
- Button "Sync-Bundle teilen" (nur sichtbar für Sources, die der User selbst angelegt hat — heuristisch über fehlenden Import-Marker)
- Button "Sync-Bundle scannen/importieren"

### Settings-Modal: neuer Tab "Sync-Sources"

- Liste der Sources (label, type, status, Edit/Delete)
- Button "Neue HTTP-Source"
- Button "Neues lokales Verzeichnis"
- Encryption-Toggle pro Source (Default ON, Toggle-OFF erfordert Bestätigungsdialog mit Warnung)

## Fehlerbehandlung

| Fehler | UX |
|---|---|
| Source unerreichbar | Indikator "error", Tooltip mit Fehlermeldung. Sync-Modal zeigt Retry-Button. |
| Auth-Fehler (401/403) | Indikator "error", Tooltip "Auth-Fehler — Sync-Bundle aktualisieren?". Settings-Tab zeigt Edit-Vorschlag. |
| Fingerprint-Mismatch | Modal-Dialog: "Source enthält Daten von anderem Stab — Sync verweigert. Source-Konfiguration prüfen oder löschen." |
| Decrypt-Fehler | Modal-Dialog: "Daten sind verschlüsselt — Passphrase passt nicht. Erneut eingeben?" |
| Schreibfehler S5 (Permission) | Modal: "Verzeichnis-Zugriff verloren — bitte neu auswählen." |
| CORS-Fehler | Modal mit Doku-Link: "Server akzeptiert CORS-Anfragen nicht. Setup-Anleitung prüfen." |

## Risiken und offene Punkte

1. **Token-Storage in localStorage:** dokumentiertes Restrisiko, akzeptiert (CSP als Mitigation)
2. **Server-Setup-Aufwand:** nginx + WebDAV ist nicht trivial für nicht-admin-versierte Customer. Doku muss ausführlich werden, ggf. Beispiel-Docker-Compose
3. **Custom-Protocol-Handler `cram://`:** funktioniert nur bei Registrierung im OS — ist nicht zuverlässig. Magic-Link-Variante deshalb sekundär; QR-Code ist primärer Distributionsweg
4. **Migration aus V1.0:** kein Migrationspfad nötig, V1.2 ergänzt nur. Bestehende Code/QR/JSON-Channels bleiben unverändert
