# V2.0 — Automatischer Online-Sync

Stand: 8. Mai 2026. Spec-Status: konvergiert, Implementierung beginnt erst nach V1.2-Praxistest.

## Beziehung zu V1.2

V2.0 baut **direkt auf der V1.2-Architektur** auf. Sync-Source-Abstraktion, S1- und S5-Backends, Encryption, Sync-Bundle, Settings-UI: bleibt unverändert. V2.0 ergänzt eine **Automatisierungs-Schicht** darüber.

Vor dem Lesen dieser Spec: [`v1.2-manual-sync.md`](v1.2-manual-sync.md) als Voraussetzung lesen.

## Ziel

"Set-and-forget"-UX. Im Krisenfall müssen die Stabsmitglieder **null** Sync-Aktionen durchführen. Sync läuft transparent. Kognitive Bandbreite bleibt für die Krise.

Konkret: ein Stand-Update auf einem Gerät erscheint auf allen anderen Geräten innerhalb von 5–10 Minuten, ohne dass jemand auf einen Knopf drückt.

## Nicht-Ziele

- Echtzeit-Sync (sub-Minute) — nicht nötig laut RC-Feedback
- WebSockets / Server-Sent-Events — Overkill für 5–10-Min-Anspruch, würde CSP weiter aufweichen
- Service-Worker-Background-Sync — komplex, Offline-Replay-Probleme
- OAuth / Cloud-API-Direktintegration — bleibt Anti-Roadmap

## User Stories (delta zu V1.2)

**US-A: Status-Eintrag wird stillschweigend verteilt.**
Bob trägt 14:23 ein, dass Carol heute nicht erreichbar ist. CRAM macht einen sofortigen Auto-Push gegen die Sync-Source. Alice hat CRAM offen und sieht spätestens 14:24:30 (nächster Polling-Tick), dass Carol als unavailable markiert ist. Toast unten rechts: "Stand aktualisiert". Kein Klick nötig.

**US-B: Konflikt wird stillschweigend aufgelöst.**
Bob und Dan markieren beide gleichzeitig (innerhalb 30 Sekunden) Status-Updates. Bobs Push gewinnt das Race (ETag-Vergleich). Dans Push schlägt mit 412 fehl. CRAM auf Dans Gerät pullt den neuen Stand, repliziert Dans Edit darauf, und pusht erneut. Beide Edits sind im finalen Stand enthalten. Dan sieht ein Toast: "Sync-Konflikt aufgelöst". Kein Dialog, keine Entscheidung von Dan gefordert.

**US-C: Tab-Wechsel triggert Soforthole.**
Alice hat CRAM in Hintergrund-Tab. Wechselt um 14:30 zu CRAM. Tab-Visibility-Event löst sofortigen Pull aus, **noch bevor** der Polling-Tick fällig wäre. Alice sieht den Stand vom letzten Push, nicht den Stand vom letzten Polling-Tick.

**US-D: Server fällt aus, Indikator alarmiert.**
Server geht 14:00 offline. Alice merkt nichts, weil sie gerade nicht editiert. Indikator wechselt von "synced" auf "syncing" (für 30 Sek), dann auf "error". Nach 5 Min Dauer-Error pulsiert der Indikator vergrößert. Alice sieht es. Wenn sie es ignoriert, ist das ein organisatorisches Problem, kein Tool-Problem (siehe Brainstorm-Konvergenz: kein Push-Alerting).

## Technische Erweiterungen

### Polling-Daemon

- **Intervall:** Default 90 Sekunden, in Settings konfigurierbar 30–300 Sekunden.
- **Implementierung:** `setInterval` auf `await source.probe()`. Wenn `serverVersion > localVersion`: `await source.read()` und State-Apply.
- **Pause-Logik:**
  - Tab versteckt → Polling auf 5 Min ausgedehnt (oder ganz ausgesetzt — TBD im Implementierung; Tab-Visibility-Event holt eh sofort wenn User zurückkehrt)
  - Browser offline (`navigator.onLine === false`) → Polling pausiert, Indikator zeigt "offline"
  - Auth-Fehler → Polling pausiert nach drei Fehlversuchen, Indikator zeigt "error"

### Auto-Push

- Trigger: **jeder lokale State-Save** (Mutation an `cram.config`, `cram.runtime.v1`, oder `cram.audit.v1`).
- Debouncing: wenn mehrere Saves innerhalb von 2 Sekunden auflaufen, wird der letzte gepusht. Verhindert Flood bei rapidem Editing.
- Backoff bei Fehler: 5s, 30s, 5min. Nach drittem Fehler: Indikator "error", User-Aktion erforderlich.

### Pull-before-Push mit If-Match

Vor jedem Push:
1. `HEAD <url>` mit `If-None-Match: <lastKnownETag>` → 304 wenn Server unverändert
2. Wenn 200 (Server hat sich geändert): erst `read()`, lokalen State mergen, dann push
3. Push: `PUT <url>` mit `If-Match: <currentETag>` → 412 wenn jemand anders zwischendurch geschrieben hat
4. Bei 412: Schritt 1 wiederholen (max 3 Iterationen, dann Toast "Sync-Konflikt — bitte prüfen")

**Merge-Strategie:** Last-Write-Wins auf gesamten State-Blob. Audit-Log-Einträge werden aber **gemerged** (deduped per Event-ID). Heißt: auch wenn ein Edit-Event den Status-Loser-Wert nicht durchsetzt, steht der Loser-Edit im Log und ist nachvollziehbar.

**Hinweis S5:** File System Access API kennt **kein** ETag/If-Match. Für S5 bleibt Last-Write-Wins ohne Konflikt-Erkennung. Im S5-Setup wird das im Settings-Tab klar angezeigt: "Diese Source unterstützt keine Konflikt-Erkennung. Bei parallelen Edits kann eine Version verloren gehen."

### Tab-Visibility-Sync

```javascript
document.addEventListener('visibilitychange', () => {
  if (document.visibilityState === 'visible') {
    syncEngine.pullAllSources();
  }
});
```

Triggert sofortigen Pull, unabhängig vom Polling-Cycle. Sehr günstig, weil Probe-Request leichtgewichtig.

### Konflikt-UX (Toast statt Dialog)

Bei eingehendem Update während User gerade etwas anderes tut:
- **Standard:** silent auto-apply, keine Anzeige
- **Wenn Update sichtbare Daten ändert** (Status einer aktuell sichtbaren Rolle, Personen-Liste sortierbar): Toast 5 Sekunden unten rechts: "Stand aktualisiert von [User] um [Zeit]"
- **Wenn lokaler Edit verloren ging** (eigenes Edit ist nicht im finalen Stand): Toast 8 Sekunden, dezent rot: "Dein Edit wurde durch eine neuere Version ersetzt — siehe Log"
- **Niemals Dialog, niemals Bestätigungs-Klick.** Das ist die harte UX-Anforderung.

### Multi-Source

- Als **Advanced-Option** in Settings sichtbar. Nicht Default.
- Bei mehreren Sources:
  - **Pull:** alle Sources parallel probieren, Source mit höchster Version gewinnt
  - **Push:** zu allen Sources schreiben (Best-Effort, einzelne Failures werden im Indikator pro Source dargestellt)
- Use-Case: Failover (primary + Standby-Server), nicht Geo-Distribution

### CSP-Erweiterung

Aktuelle CSP: `connect-src 'self' blob: data:`.

V2.0 muss `connect-src` öffnen für die konfigurierten Source-URLs. Zwei Optionen:

**Option A — generelle https-Erweiterung:**
```
connect-src 'self' blob: data: https:;
```

**Option B — explizite Domain-Allowlist mit Build-Step:**
Würde Single-File-Prinzip brechen → Anti-Roadmap.

**Entscheidung:** Option A. Kompromiss: alle HTTPS-Domains werden für `fetch()` zugelassen. Die Sicherheits-Trennung kommt aus der CRAM-Logik (nur konfigurierte Sources werden angesprochen) plus Source-Auth. Die CSP ist Sekundär-Schutz, nicht Primär-Schutz — das war sie schon vorher (Primär-Schutz ist `escapeHTML`). Die CSP-Aufweichung wird in `CHANGELOG.md` und im `SECURITY.md` (zu erstellen) explizit dokumentiert.

### Doppelte Defense

Da CSP gelockert wird:
- **Source-URL-Validation:** vor jedem fetch() wird die URL-Origin gegen die Liste der konfigurierten Sources geprüft. Verhindert versehentliches Senden an "fremde" URLs.
- **Connection-Logging im Audit:** jede Sync-Connection wird als Audit-Eintrag protokolliert (`sync_pull`, `sync_push`) mit Source-ID und Status-Code.

## Risiken und offene Punkte

1. **CSP-Aufweichung auf `https:`** ist eine Trade-off-Entscheidung. Akzeptiert, weil Single-File-Architektur keine Build-time-Domain-Allowlist erlaubt. Im SECURITY.md transparent machen.
2. **Polling-Last bei 100 Usern:** ~1 Req/Sek auf den nginx → trivial. Bei 1000 Usern → 11 Req/Sek → immer noch trivial. Bei 10000 Usern → 110 Req/Sek → würde Server-Konfiguration prüfen müssen. Ist aber außerhalb des realistischen Einsatzgebiets.
3. **Verlust-Edits trotz Audit-Log:** wenn zwei User exakt gleichzeitig den **gleichen Status-Wert** ändern, ist der Verlierer-Edit im Audit-Log nicht unterscheidbar von einem normalen Edit, weil der Endwert gleich ist. Sehr seltener Edge-Case, dokumentieren statt aufwendig lösen.
4. **Server-Maintenance-Fenster:** wenn Customer den Sync-Server neustartet, sehen alle 100 User für 30 Sek "error". Dokumentieren als erwartetes Verhalten.
