# V2.1 — Team-Pools

Stand: 20. Mai 2026, nach V2.0.0 GA. Spec-Status: **bereit für UX-Design-Schritt**, danach Implementierung.

V2.1 ergänzt die Rollen-Topologie um eine zweite Personen-Anker-Schicht: **Team-Pools**. Ein Pool ist eine benannte Gruppe von Personen, die einem Lead-Slot zugeordnet sind und in einer Krise als Pool von Fachkräften adressierbar werden — unabhängig davon, ob sie in einer Substitution-Chain stehen oder nicht.

## Beziehung zu V2.0

V2.1 baut auf dem V2.0-Datenmodell auf (Sync-Source-Abstraktion, Auto-Sync, Settings-Akkordeon bleiben unverändert). Pools sind ein neues Konfigurations-Feld. Status pro Person wird durch das bestehende Absence-Modell abgebildet — Pools erzwingen keine eigene Status-Schicht. Vor dem Lesen: [`v2.0-auto-sync.md`](v2.0-auto-sync.md) und [`v2x-print-refresh.md`](v2x-print-refresh.md) als Kontext.

UX-Layout-Varianten zu den offenen Detail-Fragen aus Sektion 12 sind ausgelagert in [`v2.1-ux-mockups.md`](v2.1-ux-mockups.md) (Patrick hat die dortigen Empfehlungen freigegeben).

## 1. Motivation und Use-Case

### Reales Problem

Krisenstäbe arbeiten mit zwei verschiedenen Personen-Strukturen, die CRAM bisher nur eine davon abbildet:

1. **Rollen-Slots mit Substitution-Chain** — funktional klar, aber starr. Ein Slot = eine aktive Person + Reservebank in fester Reihenfolge.
2. **Fach-Pools** — die Realität in jedem reifen SOC: hinter dem Lead „Head of SOC" steht ein Pool aus SOC-Analysten Tier 2/3, von denen je nach Schicht, Verfügbarkeit und Fachgebiet jemand greifbar ist. Diese Personen stehen typischerweise *nicht* alle in der Substitution-Chain des Leads, sind aber im Krisenfall die ersten, die der Lead anspricht.

Ohne Pool-Layer muss das Krisenstabs-Mitglied entweder
- alle Pool-Mitglieder als zusätzliche Slots ins Chart pressen (überfrachtet), oder
- den Pool nur im Kopf haben („ich rufe halt jemanden aus dem SOC an") — was im Stress versagt.

### Szenario

14:42 — Kompromittierungs-Vorfall, Cloud-Forensik wird benötigt. Krisenleitung fragt: „wer kann in der nächsten Stunde Azure-Forensik?"

Heute: niemand weiß es ad-hoc, Telefonkette beginnt.

Mit V2.1: Suche im Tool nach Keyword „Azure-Forensik" → drei Personen, davon eine grün (verfügbar), eine gelb (eingeschränkt), eine grau (Status unbekannt). Direkt-Kontakt-Daten sichtbar. Lead-Bezug („Head of SOC") sichtbar. Entscheidung in 20 Sekunden statt 5 Minuten.

### Abgrenzung zur Rolle

| Rolle | Pool |
|---|---|
| Genau ein Slot, eine aktiv besetzende Person | Liste von Personen ohne festen Slot |
| Substitution-Chain in fester Reihenfolge | Keine Reihenfolge, Verfügbarkeit-getrieben |
| Cascade-Visualisierung bei Vertretung | Keine Cascade allein durch Pool-Mitgliedschaft |
| Pro Level platziert | Hängt am Lead-Slot, also indirekt an dessen Level |

Pool ergänzt Rolle. Pool ersetzt Rolle nicht.

## 2. Patrick's Design-Entscheidungen (D1–D3)

Diese drei Entscheidungen sind verbindlich. Detail-Layout ist offen, Strukturentscheidungen nicht.

### D1 — Pool-Position im Layout

- Pools sind **nicht freistehend** und **kein eigener Tab**.
- Jeder Pool gehört zu einem **konkreten Lead-Slot** in einem konkreten Level.
- Pool wird **am Ende des Level-Blocks** gerendert, visuell seinem Lead zugeordnet (Linien-Verbindung, Einrückung oder geklammerte Hülle — UX-Engineer entscheidet die finale Darstellung).
- Pool darf **null, einen oder mehrere** Lead-Slots haben. Typisch eins. Im Datenmodell: ein primärer Lead, weitere als sekundäre Liste.

### D2 — Personen-Beziehung

- Person ist **global** (bestehende `persons`-Liste in `State.config`).
- Eine Person kann gleichzeitig:
  - in der Substitution-Chain einer oder mehrerer Rollen stehen,
  - Mitglied in einem oder mehreren Pools sein,
  - beides parallel.
- **Konsequenz für Cascade-Logik:** keine Änderung. Ist ein Pool-Mitglied auch Substitut in der Chain seines Leads, greift die bestehende V1.x-Cascade. Pool ist nur ein zusätzlicher Anker, keine Datenduplizierung.

### D3 — Keywords

- Freie Tags **pro Person**, nicht pro Pool.
- Auto-Vervollständigung aus bisher genutzten Tags der Config.
- Such-Filter listet alle existierenden Keywords zur Auswahl auf.
- Keine kuratierte Taxonomie. Keine Pool-Standard-Keywords.

## 3. Datenmodell

```javascript
State.config = {
  // ...alle V2.0-Felder unverändert
  schemaVersion: 3,        // war 2
  pools: [
    {
      id: 'pool-<uid>',
      name: string,                    // z.B. "SOC Tier 2/3"
      leadRoleId: string,              // ID des Lead-Slots, dem dieser Pool zugeordnet ist
      secondaryLeadRoleIds: string[],  // optional, leer wenn nur ein Lead
      memberIds: string[],             // Referenzen auf State.config.persons[i].id
      notes: string                    // optional, beschreibender Text
    }
  ]
}

State.config.persons[i] = {
  // ...alle bisherigen Felder unverändert (id, name, contact, …)
  keywords: string[]       // NEU pro Person, leerer Array erlaubt
}
```

### Schema-Migration v2 → v3

Bei V2.0-State (`schemaVersion === 2`) ohne `pools` und ohne `person.keywords`:

1. `state.config.pools = []`
2. Jede Person bekommt `keywords: []` falls Feld fehlt
3. `schemaVersion = 3`
4. Migration-Banner einblenden: „Team-Pools sind neu — leg den ersten Pool unter Settings → Pools an" (analog V2-Migration-Banner)

**Idempotenz-Trigger (Dual):** `schemaVersion < 3` ODER (`pools === undefined` && jede Person ohne `keywords`). Beide Bedingungen lösen die Migration aus, danach setzt sie sich selbst auf 3 zurück. Verhindert State-Kollisionen bei mehrfach-geöffneten Tabs während der Migration.

**Auto-Sync (V2.0):** Migration wird nach lokalem State-Save sofort gepusht (debounced 2s). Andere Geräte ziehen die migrierte Config beim nächsten Polling-Tick. Keine Sondererwartung an die Sync-Schicht — volle Konfig-Sync deckt das Feld automatisch ab.

### Verwaister Pool (Lead-Rolle gelöscht)

Wird die zugeordnete Lead-Rolle gelöscht, **bleibt der Pool erhalten**. Verhalten:

- `pool.leadRoleId = null` (statt komplett gelöscht — Pool-Daten bleiben recoverable)
- Pool wird in eine eigene „Unassigned"-Sektion gerendert (am Ende aller Levels oder als eigener UI-Bereich, UX-Engineer entscheidet)
- User kann Pool später wieder einem Lead zuweisen ODER explizit löschen
- Member bleiben unverändert (Person-Pool-Membership überlebt Lead-Delete)
- Secondary-Lead-IDs bei Lead-Delete: aus `secondaryLeadRoleIds` herausfiltern (nicht null setzen)

Rationale: Lead-Delete ist im Krisen-Alltag typischerweise eine Umstrukturierung, kein endgültiges Wegwerfen. Member-Zusammenstellungen sind teurer in der Wiederherstellung als die Lead-Zuweisung selbst.

## 4. UI-Layout

### Chart-Render-Pfad

Pro Level werden zuerst alle Rollen-Cards gerendert (bestehende Logik). Danach folgt eine **Pool-Sektion** am Ende des Level-Blocks. Die Pool-Sektion enthält alle Pools, deren `leadRoleId` zu einem Lead-Slot dieses Levels gehört.

Pro Pool:
- **Pool-Header:** Pool-Name + visueller Lead-Bezug (Linie zum Lead-Slot, Einrückung unter Lead, Klammer-Hülle oder Pfeil — **UX-Engineer entscheidet**).
- **Member-Liste:** horizontale Pill-Liste oder kompakte Listen-Zeile (flacher als Rollen-Cards). Pool und Rolle dürfen nicht visuell verwechselbar sein.
- **Pro Member:** Name + Verfügbarkeit-Indikator (grün/gelb/rot/grau, identische Farb-Codes wie Rollen-Slot) + optional „Sub für Lead"-Badge wenn Person in Substitution-Chain des Leads steht + Keyword-Tags on hover oder bei Tap.

### Member-Sort-Reihenfolge (verbindlich)

Pool-Mitglieder werden in folgender Reihenfolge dargestellt:

1. **Primär: Verfügbarkeit** (verfügbar → teilweise verfügbar → abwesend)
2. **Sekundär: Alphabetisch** nach Person-Name (innerhalb jeder Verfügbarkeits-Gruppe)

Begründung: im Krisenfall sucht der Stab zuerst „wer ist überhaupt erreichbar". Alphabetisch dient nur als sekundärer Stabilitäts-Anker, damit die Reihenfolge nicht zufällig wirkt zwischen Renders.

### Pool-CRUD

Über **Settings → neuer Tab „Pools"**:
- Liste aller Pools mit Lead-Spalte und Member-Count.
- Add/Edit-Modal: Felder Name, Lead-Auswahl (Picker aus existierenden Slots), sekundäre Leads (optional Multi-Select), Member-Picker (Multi-Select aus `persons`), Notes.
- Delete: bestätigender Dialog, Hinweis dass Personen erhalten bleiben (nur Pool-Verknüpfung wird gelöscht).

### Keywords-UI

In bestehendem **Person-Edit-Modal**: neues Feld „Schwerpunkte" als Tag-Input mit Autocomplete. Vorschlagsliste aus `State.config.persons.flatMap(p => p.keywords)` dedupliziert.

## 5. Cascade-Auswirkung

Pool-Existenz allein ändert keine Cascade-Berechnung. Nur die Substitution-Chain des Leads zählt für Cascade-Render (`detectCascades()` bleibt unverändert).

Was sich ändert: **Pool-Member-Rendering**. Bei Pool-Render-Zeit:
- Prüfe pro Member, ob er in der Chain des `leadRoleId`-Slots steht und ob er gerade aktiv ist (Cascade-Quelle oder -Ziel).
- Wenn ja: Member-Pill bekommt „Sub aktiv"-Badge oder Akzentfarbe (UX-Engineer entscheidet).

Keine zusätzliche Cascade-Berechnung, nur Anzeige-Anreicherung. Das hält die V1.x-Cascade-Logik strikt vom Pool-Layer getrennt.

## 6. Such- und Filter-UI

### Funktion

- Globale Suche nach Keyword: zeigt alle Personen mit dem Tag.
- Filter-Kombination: Keyword + Verfügbarkeits-Status (z.B. „Cloud-Forensik" + nur grün).
- Treffer-Liste zeigt: Name, Pool-Zugehörigkeit, Lead-Bezug, Verfügbarkeit, Direkt-Kontakt.

### Position

Sidebar-Tab (analog People-Tab) oder Header-Action neben „Print" / „Sync" — **UX-Engineer entscheidet**. Bedingung: maximal zwei Klicks vom Chart bis zur Treffer-Liste, weil Krisen-Use-Case zeitkritisch ist.

### Datenquelle

Nur lokale `State.config.persons[].keywords`. Keine Netz-Calls, keine Server-Logik. Volltext über Tags genügt — kein Fuzzy-Match, kein Stemming.

## 7. Print-Integration

V2.1 ändert die Render-Topologie pro Level (Pool-Sektion am Ende). Damit ist ein **Print-Refresh ohnehin nötig**, weil die bestehenden Print-Templates eine flache Rollen-Liste pro Level annehmen. Print-Refresh und V2.1 werden deshalb gemeinsam umgesetzt.

Verweis: [`v2x-print-refresh.md`](v2x-print-refresh.md). Konkrete Anforderungen aus V2.1:

- **Overview-Template:** pro Lead-Slot Pool-Erwähnung als kompakte Sub-Zeile (z.B. „SOC Tier 2/3 — 4 Mitglieder"). Nicht jede Person einzeln im Overview.
- **Detail-Template:** pro Pool ein eigener Abschnitt unter dem zugehörigen Lead, mit voller Member-Liste, Kontaktdaten und Keywords.
- **People-Template:** Keywords pro Person als zusätzliche Spalte oder Zeile. Sortier-Option nach Pool-Zugehörigkeit ergänzen.

## 9. Implementierungs-Reihenfolge (verbindlich)

1. **Schema-Migration v2 → v3**: `pools: []` und `person.keywords: []` ergänzen, Migration-Banner-Trigger, Dual-Idempotenz.
2. **Datenmodell-Helper im Storage-IIFE**: Pool-CRUD (`createPool`, `updatePool`, `deletePool`, `addMemberToPool`, `removeMemberFromPool`), Keywords-Helper (`addKeyword`, `removeKeyword`, `listAllKeywords`).
3. **Settings-UI**: neuer Tab „Pools", Pool-Liste, Pool-Edit-Modal mit Lead- und Member-Picker.
4. **Person-Edit-Modal**: Keywords-Field mit Tag-Input und Autocomplete.
5. **Chart-Render**: Pool-Sektion pro Level (am Ende, visuelle Zuordnung zum Lead). Member-Pills mit Verfügbarkeit-Indikator.
6. **Such-/Filter-UI**: Position Sidebar oder Header (UX-entschieden), Treffer-Liste mit Keyword + Status-Filter.
7. **Print-Refresh** gemeinsam: `v2x-print-refresh.md` umsetzen, Pool-spezifische Anforderungen aus Sektion 7 integrieren.
8. **i18n × 5 Sprachen** ergänzen.
9. **Mobile-Pass**: 44pt Touch-Targets auf Pool-Pills, Safe-Area-Insets prüfen, Portrait/Landscape testen.

Schritte 1–4 sind voneinander unabhängig nach Schritt 1. Schritt 5 setzt 2 voraus. Schritt 6 setzt 4 voraus. Schritt 7 setzt 5 voraus. Schritt 8 kann parallel ab Schritt 3 laufen.

## 10. Akzeptanzkriterien

- [ ] Pool wird ausschließlich in `localStorage` (`cram.config`) gespeichert. Keine externe Persistenz, keine neuen Storage-Keys.
- [ ] Migration v2 → v3 ist idempotent (Dual-Trigger: `schemaVersion` und Feld-Presence).
- [ ] Pool-Render funktioniert offline (keine Netz-Calls).
- [ ] Auto-Sync (V2.0) überträgt Pool-Daten korrekt — volle Konfig-Sync, kein separater Status-Sync für Pools.
- [ ] Eine Person in 2 Pools + 3 Rollen-Chains funktioniert ohne Duplikation und ohne Inkonsistenz im Audit-Log.
- [ ] Mobile: Pool-Member-Tap-Area ≥ 44 × 44 pt.
- [ ] Print-Templates rendern Pool-Inhalt sinnvoll: nicht abgeschnitten, keine leeren Spalten, lesbar auf A4.
- [ ] Pool ohne Member rendert ohne Fehler (leere Pill-Liste, Hinweis „noch keine Mitglieder").
- [ ] Pool-Delete entfernt nur Pool-Eintrag, Personen bleiben in `State.config.persons` erhalten.
- [ ] Schema-Migration löst Migration-Banner aus, der genau einmal pro Gerät erscheint (`cram.prefs.v1` merkt den Acknowledgement).
- [ ] CSP-Smoke-Test besteht (keine `unsafe-eval`-Notwendigkeit, keine externen Ressourcen).
- [ ] XSS-Schutz: Pool-Name, Keywords und Notes laufen durch `escapeHTML()` an jeder Interpolations-Stelle.

## 11. Was bewusst NICHT in V2.1

- Pool-übergreifende Skills- oder Zertifikate-Verwaltung — Pools kennen nur freie Keywords, keine strukturierte Skill-Matrix.
- Server-vermittelte Pool-Sync-Konflikte mit eigener Konflikt-UI — gleicher ETag-Mechanismus wie V2.0, gleiche Last-Write-Wins-Logik.
- Externe HR-System-Integration (LDAP-Anbindung, AD-Sync, BambooHR etc.).
- Bereitschafts-Kalender / Schicht-Voraus-Planung — kommt in V2.2.
- Pool-Member-Statuszeit-Voraus-Planung („Anna ab 16:00 verfügbar") — auch V2.2-Kalender.
- Übungs-Modus / Tabletop-Exercise-Mode.
- Pool-Templates oder Pool-Bibliothek („Standard-SOC-Pool importieren").
- Berechtigungs-Modell („nur SOC-Lead darf SOC-Pool editieren") — V2.x bleibt single-tenant ohne RBAC.

## 11a. Sync-Verhalten bei Pool-/Keyword-Änderungen

V2.0 hat einen Sync/Data-Split: Status-Mode (Verfügbarkeit) wird gegen einen Config-Fingerprint validiert, Daten-Mode (Konfiguration) explizit angefordert. Der Fingerprint wird über die Personen-IDs berechnet (`computeConfigHash`), **NICHT** über Pools oder Keywords.

**Konsequenz:**

- Pool-Hinzufügen, Pool-Edit, Member-Änderungen, Keyword-Änderungen **triggern keinen** Status-Drift.
- Sie werden vom **Daten-Mode-Sync** mitgesendet, ähnlich wie Rollen-/Personen-Konfiguration.
- Beim Auto-Sync (V2.0) bedeutet das: ein Status-only Auto-Sync zieht keine Pool-Daten aktualisiert.

**Best Practice:**

- Nach Pool-/Keyword-Änderungen einen manuellen Daten-Pull/Push triggern, wenn der lokale Stab eine zweite Instanz hat.
- Oder den Sync-Source auf `bidirectional` umstellen mit kurzem Polling-Intervall — Daten-Mode-Differenzen werden dann erkannt.

Bewusste Konsequenz des V1.3-Sync/Data-Split-Designs. Pool ist Konfigurationsmaterial, nicht Stab-Status.

## 12. Offene Fragen für UX-Engineer (vor Implementierung)

1. **Visuelle Lead↔Pool-Zuordnung:** Linie vom Lead-Slot zum Pool-Header, eingerückte Position unter Lead, Klammer-Hülle, gemeinsamer Farb-Akzent — welche Variante kommuniziert die Zuordnung am eindeutigsten, ohne die Rollen-Card-Lesbarkeit zu stören?
2. **Pool-Member-Pill-Größe:** wie viel flacher als Rollen-Cards? Konkrete Pixel- und mm-Maße für Bildschirm und Print.
3. **„Sub für Lead"-Badge auf Pool-Member:** Position (oben rechts, unterhalb des Namens, als Rahmen-Farbe), Wording, Farbcode — und Unterschied zur Cascade-Pfeil-Codierung im Rollen-Layer (Verwechslungsgefahr).
4. **Such-UI-Position:** Sidebar-Tab oder Header-Action? Argumente für jede Seite, Empfehlung mit Begründung.
5. **Pool-Edit-Modal-Layout:** 1-Spalte (gestapelt: Name → Lead → Members → Notes) oder 2-Spalte (Lead-Auswahl + Member-Picker nebeneinander)? Welche Variante reduziert Scroll-Aufwand bei großen Member-Listen?
6. **Mobile-Darstellung Pool-Sektion:** stapeln unter dem Level, eigene Akkordeon-Sektion oder horizontaler Scroll-Container? Krise auf iPhone — was bleibt lesbar?
7. **Keyword-Tag-Visualisierung:** Tooltip on hover, dauerhaft sichtbar als kleine Chips unter dem Namen, oder nur in der Such-Trefferliste? Trade-off zwischen Informationsdichte und Übersichtlichkeit.
