## Q1 — Visuelle Lead↔Pool-Zuordnung

Aktuell rendert das Chart pro Level einen horizontalen Block aus Rollen-Cards. V2.1 hängt am Ende dieses Blocks einen Pool, der laut `pool.leadRoleId` einem konkreten Lead-Slot dieses Levels gehört. Die Beziehung muss visuell erkennbar sein, ohne die Rollen-Card-Lesbarkeit zu stören.

### Variante (a) — Einrückung mit vertikaler Linie

```text
Level 2 — Operative Leitung
+---------------------+   +---------------------+   +---------------------+
| Head of SOC         |   | Forensics Lead      |   | Legal Counsel       |
| Dr. M. Bachmeier    |   | M. Lefèvre          |   | E. Whitfield        |
| [active]            |   | [active]            |   | [active]            |
+---------------------+   +---------------------+   +---------------------+
   |
   +---- SOC Tier 2/3 ............................................
         ( S. Köhler  F. Öztürk  M. Großmann  E. Süßmann )
```

**Vorteil:** Klar lesbar — Linie führt vom Lead-Card-unteren-Rand zum Pool. Bekanntes Org-Chart-Pattern. Funktioniert auch monochrom.
**Nachteil:** Bei mehreren Pools pro Level muss die Linie zwischen Cards durchgezogen werden, kann unruhig wirken. Wenn der Lead-Slot nicht der linkeste im Level ist, wird die Linie schräg oder muss umgeleitet werden.

### Variante (b) — Klammer-Hülle mit Lead-Bezug-Label

```text
Level 2 — Operative Leitung
+---------------------+   +---------------------+   +---------------------+
| Head of SOC         |   | Forensics Lead      |   | Legal Counsel       |
| Dr. M. Bachmeier    |   | M. Lefèvre          |   | E. Whitfield        |
+---------------------+   +---------------------+   +---------------------+

  pool of: Head of SOC
+--------------------------------------------------------------------+
| SOC Tier 2/3                                                       |
|   S. Köhler   F. Öztürk   M. Großmann   E. Süßmann                 |
+--------------------------------------------------------------------+
```

**Vorteil:** Eindeutig durch Text-Label „pool of: <Lead-Name>". Funktioniert unabhängig von der Position des Leads im Level. Pool-Box ist visuell vom Rollen-Card-Layer getrennt (andere Border-Stärke/Hintergrund).
**Nachteil:** Mehr Vertikal-Platz. Bei sehr langen Lead-Rollennamen wird das Label lang. Pool wirkt eher „neben" als „unter" dem Lead.

### Variante (c) — Pool nimmt Lead-Border-Farbe + Lead-Slot trägt Pool-Icon

```text
Level 2 — Operative Leitung
+=====================+   +---------------------+   +---------------------+
| Head of SOC      [P]|   | Forensics Lead      |   | Legal Counsel       |
| Dr. M. Bachmeier    |   | M. Lefèvre          |   | E. Whitfield        |
+=====================+   +---------------------+   +---------------------+
   (Lead-Card-Border in Akzent-Farbe, [P] = Pool-Indicator-Icon)

+====================================================================+
| SOC Tier 2/3                                                       |
|   S. Köhler   F. Öztürk   M. Großmann   E. Süßmann                 |
+====================================================================+
   (gleiche Akzent-Farbe wie Lead-Border)
```

**Vorteil:** Sehr kompakt. Lead-Slot zeigt Pool-Existenz auch ohne ans Level-Ende zu scrollen.
**Nachteil:** Verletzt CRAM-Prinzip „color is not the signal" wenn das `[P]`-Icon klein/weglassbar ist. Bei mehreren Pools pro Level müsste jeder eine eigene Akzent-Farbe haben — skaliert nicht. Konflikt mit bestehender Cascade-Farb-Codierung (rot/gelb).

### Variante (d) — Eigene Zeile am Level-Ende mit Lead-Name als Header

```text
Level 2 — Operative Leitung
+---------------------+   +---------------------+   +---------------------+
| Head of SOC         |   | Forensics Lead      |   | Legal Counsel       |
| Dr. M. Bachmeier    |   | M. Lefèvre          |   | E. Whitfield        |
+---------------------+   +---------------------+   +---------------------+

Pools                                                                       
  >> Head of SOC — SOC Tier 2/3
     S. Köhler   F. Öztürk   M. Großmann   E. Süßmann
  >> Forensics Lead — DFIR Bench
     A. Rousseau   T. Mercier
```

**Vorteil:** Skaliert auf beliebig viele Pools pro Level. Erträglich auf Mobile (linear gestapelt). Text-First, kein Linien-Routing nötig.
**Nachteil:** Die räumliche „dieser Pool gehört zu Lead X"-Information verschwindet — sie wird nur noch über das Text-Label kommuniziert. Optisch schwächere Bindung.

### **EMPFEHLUNG:** Variante (b) — Klammer-Hülle mit Lead-Bezug-Label

**Begründung:** Funktioniert unabhängig von Lead-Position im Level, skaliert auf mehrere Pools, kein Linien-Routing-Problem, und das Text-Label macht die Beziehung sprachlich eindeutig — Farb-Codierung bleibt frei für Verfügbarkeits-Status. Die Box-Border und der Hintergrund sorgen für den klaren visuellen Layer-Unterschied zu Rollen-Cards. Variante (a) wäre die zweite Wahl, kollidiert aber mit dem Layout-Engine-Aufwand bei Pools, deren Lead nicht der linkeste Slot ist.

## Q2 — Pool-Member-Pill-Größe vs. Rollen-Card

Rollen-Cards sind groß (Name, Aktive Person, Status, Substitut-Hinweis). Pool-Member sind weniger informationsreich pro Stück — und Pools haben oft 4–10 Members. Sie müssen visuell „flacher" sein als Cards, sonst geht der Layer-Unterschied verloren.

### Variante (a) — Pills: Single-Line, Name + kleines Status-Icon, max 4 pro Reihe

```text
+--------------------------------------------------------------------+
| SOC Tier 2/3                            pool of: Head of SOC       |
|  ( o S. Köhler )  ( o F. Öztürk )  ( o M. Großmann )  ( ! E. Süßmann ) 
|  ( - U. Kägi )    ( o A. Rousseau )                                |
+--------------------------------------------------------------------+
   (o = available / ! = limited / - = unknown / x = unavailable)
```

**Vorteil:** Sehr dicht, viele Members auf einem Blick. Status-Icon + Form (rund vs eckig optional) statt nur Farbe.
**Nachteil:** Bei sehr langen Namen („Jennifer Martinez-Klein") laufen Pills über. Status-Icon braucht echte Glyphen (nicht ASCII-`o`/`!`), damit es nicht wie Tippfehler aussieht.

### Variante (b) — Mini-Cards: 2-zeilig, Name + Verfügbarkeits-Status, max 3 pro Reihe

```text
+--------------------------------------------------------------------+
| SOC Tier 2/3                            pool of: Head of SOC       |
|  +-------------------+ +-------------------+ +-------------------+ |
|  | S. Köhler         | | F. Öztürk         | | M. Großmann       | |
|  | available         | | available         | | available         | |
|  +-------------------+ +-------------------+ +-------------------+ |
|  +-------------------+ +-------------------+                       |
|  | E. Süßmann        | | U. Kägi           |                       |
|  | limited           | | unknown           |                       |
|  +-------------------+ +-------------------+                       |
+--------------------------------------------------------------------+
```

**Vorteil:** Status als Text, nicht nur Icon — voll lesbar ohne Farben. Pro Mini-Card mehr Platz, lange Namen passen.
**Nachteil:** Mini-Cards sehen Rollen-Cards zu ähnlich, der visuelle Layer-Unterschied wird schwächer. Pro Pool deutlich mehr Vertikal-Platz.

### Variante (c) — Listen-Layout: vertikale Liste, Name links, Status-Pill rechts

```text
+--------------------------------------------------------------------+
| SOC Tier 2/3                            pool of: Head of SOC       |
|  S. Köhler                                       [ available ]     |
|  F. Öztürk                                       [ available ]     |
|  M. Großmann                                     [ available ]     |
|  E. Süßmann                                      [ limited   ]     |
|  U. Kägi                                         [ unknown   ]     |
+--------------------------------------------------------------------+
```

**Vorteil:** Sehr lesbar, scannbar wie eine Tabelle. Lange Namen kein Problem. Status-Pill ist textuell + farblich.
**Nachteil:** Vertikal-lastig — bei 10 Members ist die Pool-Sektion länger als das ganze Level oben. Wirkt eher wie Listen-Tab statt Chart-Element.

### **EMPFEHLUNG:** Variante (a) — Pills mit Status-Icon + Form, max 4 pro Reihe

**Begründung:** Erfüllt das „klar flacher als Rollen-Cards"-Ziel und hält den Vertikalplatz im Griff. Color-is-not-the-signal-Prinzip durch Icon-Pair erfüllt. Bei sehr langen Namen kann das Pill auf maximal sinnvolle Breite (z.B. 22 char) gekürzt werden mit Hover-Tooltip — die volle Identifikation läuft sowieso über Tap-/Click → Detail-Modal. Variante (c) wäre die Mobile-Default (siehe Q6), aber Desktop bleibt Pills.

## Q3 — „Sub für Lead"-Badge auf Pool-Member

Wenn ein Pool-Member parallel in der Substitution-Chain des Pool-Leads steht, ist das relevante Information („dieser SOC-Analyst ist auch offizieller Vertreter des Head of SOC"). Wie kennzeichnen, ohne mit dem Cascade-Pfeil-Layer zu kollidieren?

### Variante (a) — Kleines „SUB"-Badge rechts neben dem Namen

```text
( o S. Köhler  [SUB] )  ( o F. Öztürk )  ( o M. Großmann )  ( ! E. Süßmann )
```

**Vorteil:** Sehr explizit, Text-Label leicht zu lernen. Keine Farb-Codierung-Verwirrung mit Cascade.
**Nachteil:** Pill wird breiter, weniger Members pro Reihe.

### Variante (b) — Farbiger Border-Left an der Pool-Member-Pill

```text
|( o S. Köhler )  ( o F. Öztürk )  ( o M. Großmann )  ( ! E. Süßmann )
^
border-left in Akzent-Farbe (z.B. blau, NICHT rot/gelb)
```

**Vorteil:** Spart Platz, Pill bleibt schmal. Visuell subtil, stört nicht.
**Nachteil:** Farbe allein ist Signal — verletzt CRAM-Prinzip. Bei Color-Blindness oder monochrom-Print verschwindet die Information.

### Variante (c) — Eigene Sub-Reihe „Substitutes" innerhalb der Pool-Box

```text
+--------------------------------------------------------------------+
| SOC Tier 2/3                            pool of: Head of SOC       |
|  Members:                                                          |
|  ( o F. Öztürk )  ( o M. Großmann )  ( ! E. Süßmann )  ( - U. Kägi )
|  Substitutes for Head of SOC:                                      |
|  ( o S. Köhler )  ( o A. Rousseau )                                |
+--------------------------------------------------------------------+
```

**Vorteil:** Maximal eindeutig durch räumliche Trennung. Wer beides ist, taucht nur in der Sub-Reihe auf — keine Dopplung.
**Nachteil:** Zwei Reihen pro Pool kostet Platz. Information „diese Person ist auch Pool-Member" ist erst aus der Reihen-Zugehörigkeit ableitbar.

### **EMPFEHLUNG:** Variante (a) — `[SUB]`-Badge an der Pill

**Begründung:** Text-Badge erfüllt das „color is not the signal"-Prinzip aus CLAUDE.md. Der Badge ist klein, klar, und in der Übersetzungs-Tabelle der Spec bereits angelegt („Vertretung aktiv" / „Substitute active"). Variante (c) ist eleganter, aber doppelter Platzverbrauch lohnt sich für eine Randinformation nicht. Optional kann zusätzlich der Pill-Border etwas dicker werden — die Farbe ist dann redundante Verstärkung, nicht alleinerklärendes Signal.

## Q4 — Such-UI-Position

Globale Suche „wer kann Keyword X" muss in der Krise maximal zwei Klicks vom Chart entfernt sein.

### Variante (a) — Neuer Sidebar-Tab „Search"

```text
Sidebar:
+------------+------------+------------+------------+
|  Roster    |  People    |  Search    |   Log      |
+------------+------------+------------+------------+
                          ^^^^^^^^^^^^^
                          NEU

Klick auf "Search":
+----------------------------------------------------+
| Search focus area                                  |
|  [ azure-forensik________________________ ]  ( v ) |
|  Filter: ( ) only available  ( ) all              |
|                                                    |
|  3 matches                                         |
|  - S. Köhler        [ available ]  Head of SOC pool|
|  - A. Rousseau      [ limited  ]   DFIR Bench pool |
|  - F. Öztürk        [ available ]  Head of SOC pool|
+----------------------------------------------------+
```

**Vorteil:** Eigener Raum, klar adressierbar, ein Klick vom Chart. Konsistent mit dem etablierten Sidebar-Tab-Muster (Roster/People/Log).
**Nachteil:** Sidebar wird breiter / Tabs werden enger. Vier Tabs statt drei — Tab-Bar wird auf Mobile knapp.

### Variante (b) — Header-Action: Such-Button öffnet Modal

```text
Header:
+----------------------------------------------------------------+
| CRAM   awareness: green  ...  [ Sync ] [ Search ] [ Settings ] |
+----------------------------------------------------------------+
                                          ^^^^^^^^^^
                                          NEU

Klick auf "Search" → Modal-Overlay:
+================ Search focus area ================+
|  [ azure-forensik________________________ ]  (v) |
|  Filter: only available [ ]    all status [ x ]  |
|                                                  |
|  3 matches                                       |
|  ...                                             |
|                              [ Close ]           |
+==================================================+
```

**Vorteil:** Modal-Overlay nimmt 100% Aufmerksamkeit — gut im Stress. Funktioniert identisch auf Mobile und Desktop. Bestehende Sidebar bleibt unverändert.
**Nachteil:** Modal blockiert das Chart darunter — Such-Treffer-Liste und Pool-Render lassen sich nicht parallel betrachten. Zwei Klicks: Open + Type.

### Variante (c) — Search-Box in People-Tab integriert

```text
Sidebar — People-Tab:
+----------------------------------------------------+
| People                                             |
|  [ search by name or focus area ____________ ] (v) |
|  Filter: focus area  [ azure-forensik     v ]      |
|          status      [ all                 v ]     |
|                                                    |
|  Dr. M. Bachmeier              Head of SOC        |
|  S. Köhler             [SUB]   Head of SOC pool   |
|  ...                                               |
+----------------------------------------------------+
```

**Vorteil:** Kein neuer Tab, kein neues UI-Element — angereicherter People-Tab. Synergie: Name-Suche + Keyword-Suche im selben Feld.
**Nachteil:** People-Tab muss heute schon mit langen Listen klarkommen — Filter und Such-State vermischen sich. Suche fühlt sich wie ein Sekundär-Feature an, nicht wie ein primärer Krisen-Pfad.

### **EMPFEHLUNG:** Variante (a) — Sidebar-Tab „Search"

**Begründung:** Erfüllt die „max zwei Klicks"-Anforderung mit nur einem Klick (Tab-Switch). Bleibt sichtbar während andere Tabs offen sind — Such-Treffer und Chart lassen sich parallel scannen. Die Sidebar-Tab-Konvention ist im Tool etabliert (User wissen, wo Sidebar-Tabs leben). Auf Mobile-Engpass kann der Tab-Bar ein Scroll-Container werden, oder Roster/People/Search/Log auf Icons reduziert mit Tooltip — beides verträglich. Variante (b) ist die Backup-Option falls Sidebar-Breite auf Mobile knapp wird.

## Q5 — Pool-Edit-Modal-Layout

Felder: Name (string), Lead-Auswahl (Dropdown aus existierenden Lead-Slots), optional Secondary Leads (Multi-Select), Member-Picker (Multi-Select aus `persons`), Notes (Textarea).

### Variante (a) — 1-spaltig, vertikal gestapelt

```text
+============= Edit Pool ==============+
| Name:                                |
|  [ SOC Tier 2/3 _________________ ]  |
|                                      |
| Lead:                                |
|  [ Head of SOC                 (v) ] |
|                                      |
| Secondary Leads (optional):          |
|  [ + add another lead          (v) ] |
|                                      |
| Members:                             |
|  +------------------------------+    |
|  | [x] S. Köhler                |    |
|  | [x] F. Öztürk                |    |
|  | [x] M. Großmann              |    |
|  | [ ] U. Kägi                  |    |
|  | [ ] A. Rousseau              |    |
|  | [ ] M. Lefèvre               |    |
|  | ...                          |    |
|  +------------------------------+    |
|                                      |
| Notes:                               |
|  [                              ]    |
|  [                              ]    |
|                                      |
|         [ Cancel ]   [ Save ]        |
+======================================+
```

**Vorteil:** Linearer Lese-Fluss von oben nach unten. Funktioniert auf Mobile direkt (keine Layout-Änderung nötig). Einfach umzusetzen.
**Nachteil:** Bei 30+ Personen wird der Member-Picker hoch, viel Scroll. Notes verschwindet unter dem Falt-Knick.

### Variante (b) — 2-spaltig: Meta links, Member-Picker rechts

```text
+================= Edit Pool ====================+
| Name:                  | Members:              |
| [ SOC Tier 2/3 ____ ]  |  [ search ________ ]  |
|                        |  +------------------+ |
| Lead:                  |  | [x] S. Köhler    | |

| [ + add lead     (v) ] |  | [ ] A. Rousseau  | |
|                        |  | [ ] M. Lefèvre   | |
| Notes:                 |  | [ ] T. Mercier   | |
| [                   ]  |  | [ ] E. Whitfield | |
| [                   ]  |  | ...              | |
| [                   ]  |  +------------------+ |
|                        |  5 selected           |
|        [ Cancel ]      [ Save ]                |

```

**Vorteil:** Member-Picker bekommt eine richtige Spalte, scrollbar ohne dass Meta verschwindet. Selection-Count immer sichtbar.
**Nachteil:** Mobile-Layout muss aufs Stapeln umfallen — das heißt zwei Render-Pfade im Code. Bei wenigen Members wirkt die rechte Spalte zu groß.

### Variante (c) — Tabs im Modal: „Basics" / „Members" / „Advanced"

```text
+============= Edit Pool: SOC Tier 2/3 ==============+
| ( Basics )  ( Members )  ( Advanced )              |
+----------------------------------------------------+
| Name: [ SOC Tier 2/3 _____________________ ]       |
|                                                    |
| Lead: [ Head of SOC                       (v) ]    |
|                                                    |
|                                                    |
|                                                    |
|              [ Cancel ]   [ Save ]                 |
+====================================================+
```

**Vorteil:** Jeder Tab klein und übersichtlich. Skaliert für zukünftige Felder.
**Nachteil:** Drei Klicks statt einem Scroll — höhere kognitive Last. Member-Auswahl und Lead-Auswahl sind in unterschiedlichen Tabs, obwohl sie konzeptuell zusammengehören. Overkill für 5 Felder.

### **EMPFEHLUNG:** Variante (a) — 1-spaltig vertikal gestapelt

**Begründung:** Konsistent mit allen anderen Edit-Modals in CRAM (Person-Edit, Role-Edit). Ein Render-Pfad für Mobile und Desktop. Bei großen Member-Listen löst eine Such-Box am Anfang des Pickers + eine `[x] all selected (5 of 32)`-Indikator-Zeile das Scroll-Problem ausreichend. Variante (b) gewinnt erst bei 50+ Personen — eine Pool-Größe, die ohnehin in Frage gestellt werden muss („ist das wirklich ein Pool oder eine Abteilung?").

## Q6 — Mobile Pool-Sektion-Darstellung (iPhone Portrait 375×667)

Auf Mobile ist Vertikal-Scroll erwartet, aber der Pool muss zwischen Level-Block und nächstem Level sinnvoll passen — ohne dass User durch riesige Pool-Boxen zum nächsten Level scrollen müssen.

### Variante (a) — Pool kollabiert per Default, expandierbar mit Tap

```text
iPhone Portrait
+--------------------------------+
| Level 2 - Operative Leitung    |
|                                |
| +----------------------------+ |
| | Head of SOC                | |
| | Dr. M. Bachmeier           | |
| | [ active ]                 | |
| +----------------------------+ |
| +----------------------------+ |
| | Forensics Lead             | |
| | M. Lefèvre                 | |
| +----------------------------+ |
|                                |
| Pools (2)                  v   |
|  > SOC Tier 2/3   (4 members)  |
|  > DFIR Bench     (2 members)  |
+--------------------------------+
```

**Vorteil:** Pools sind in der Liste sichtbar (Pool-Existenz nicht versteckt), aber stören den Chart-Scroll nicht. Tap → Expand zeigt Members. Sparsamster Vertikal-Verbrauch.
**Nachteil:** Zusätzlicher Tap pro Pool, um Members zu sehen. Im Krisenfall ist Tappen unter Stress fehleranfälliger als Scrollen.

### Variante (b) — Pool immer expandiert, Member-Pills brechen mehrzeilig

```text
iPhone Portrait
+--------------------------------+
| Level 2 - Operative Leitung    |
|                                |
| +----------------------------+ |
| | Head of SOC                | |
| | Dr. M. Bachmeier           | |
| +----------------------------+ |
| +----------------------------+ |
| | Forensics Lead             | |
| | M. Lefèvre                 | |
| +----------------------------+ |
|                                |
| +----------------------------+ |
| | SOC Tier 2/3               | |
| | pool of: Head of SOC       | |
| | ( o S. Köhler )            | |
| | ( o F. Öztürk )            | |
| | ( o M. Großmann )          | |
| | ( ! E. Süßmann )           | |
| +----------------------------+ |
+--------------------------------+
```

**Vorteil:** Members immer sichtbar, kein zusätzlicher Tap. Lange Liste = mehr Scroll, aber Scroll ist auf Mobile billiger als Tap+Expand.
**Nachteil:** Bei 4 Pools à 8 Members pro Level wird der Block riesig. Nächster Level rutscht weit nach unten — Krisen-Überblick leidet.

### Variante (c) — Pool als eigener Akkordeon-Eintrag mit Lead-Name als Header

```text
iPhone Portrait
+--------------------------------+
| Level 2 - Operative Leitung    |
| +----------------------------+ |
| | Head of SOC                | |
| | Dr. M. Bachmeier           | |
| +----------------------------+ |
| +----------------------------+ |
| | Forensics Lead             | |
| +----------------------------+ |
+--------------------------------+
+--------------------------------+
| > Pool: Head of SOC        (4) |
| > Pool: Forensics Lead     (2) |
+--------------------------------+
+--------------------------------+
| Level 3 - ...                  |
+--------------------------------+
```

**Vorteil:** Pools sind klar zwischen Levels, nicht „im" Level — kompakter Chart-Scroll. Akkordeon-Pattern ist auf Mobile etabliert.
**Nachteil:** Pool ist räumlich vom Level-Block getrennt — die Zuordnung „dieser Pool gehört zum Lead in Level 2" muss komplett über das Text-Label transportiert werden. Schwächt D1.

### **EMPFEHLUNG:** Variante (a) — Pool kollabiert per Default

**Begründung:** Erhält die Pool-am-Ende-des-Level-Blocks-Zuordnung aus D1. Erlaubt dem Krisenstabs-Mitglied, im Überblick zu bleiben und Pools nur dann zu expandieren, wenn der Pool relevant wird. Ein Pref-Toggle „Pools immer expandiert" kann später ergänzt werden (V2.x). Tap-Target für den Pool-Header ist groß genug für die 44pt-Anforderung. Variante (b) ist die richtige Default für Desktop (siehe Q1+Q2).

## Q7 — Keyword-Tag-Visualisierung

Zwei Sub-Fragen: (i) wie sieht das Keyword-Tag-Input im Person-Edit-Modal aus? (ii) wo zeigt sich das Keyword einer Person im Chart-Render?

### Q7-i: Tag-Input im Person-Edit-Modal

#### Variante (a) — Chip-Style mit `×`-Button

```text
Focus areas:
  [ azure-forensik  x ]  [ incident-response  x ]  [ malware-analysis  x ]
  [ add focus area ____________________ ]
```

**Vorteil:** Direkt-manipulation: Chip = ein Datensatz, `×` entfernt ihn. Visuell klar trennbar von Free-Text.
**Nachteil:** Auto-Vervollständigung muss am Input-Feld unten hängen — Render-Logik komplexer.

#### Variante (b) — Comma-separated Text-Input mit Dropdown-Vorschlag

```text
Focus areas:
  [ azure-forensik, incident-response, malware-analysis|___ ]
                                                       v
                                                       Suggestions:
                                                       - malware-reverse
                                                       - mobile-forensik
```

**Vorteil:** Sehr einfach umzusetzen. Tastatur-only-Workflow ohne Maus.
**Nachteil:** Komma als Trenn-Zeichen ist fehleranfällig (Vergessen, Tippfehler). Entfernen eines Tags = Cursor-Navigation. Keine visuelle Bestätigung welche Tags „aktiv" sind.

#### Variante (c) — Hybrid: Chips oben + Text-Input mit Autocomplete unten

```text
Focus areas:
  [ azure-forensik  x ]  [ incident-response  x ]  [ malware-analysis  x ]

  Add focus area:
  [ mob|_______________ ]
        v
        - mobile-forensik
        - mobile-incident-response
```

**Vorteil:** Beide Welten: existierende Tags visuell als Chips, neue über Input mit Autocomplete. Klare Trennung „was ist gesetzt" vs „was füge ich hinzu".
**Nachteil:** Zwei UI-Elemente statt einem. Etwas mehr Vertikal-Platz.

#### **EMPFEHLUNG für Q7-i:** Variante (c) — Hybrid Chips + Autocomplete-Input

**Begründung:** Macht den Unterschied zwischen „setzen" und „lesen" explizit. Autocomplete erfüllt D3 (Vorschläge aus existierenden Tags) ohne dass User die existierende Liste raten muss. Chip-`×` entfernt sicher per Klick, ohne Komma-Editier-Risiko. Mobile-tauglich: Chip-Tap zum Entfernen ist ein klares 44pt-Target.

### Q7-ii: Keyword-Anzeige im Chart-Render

#### Variante (a) — Permanent als Mini-Badge unter dem Namen

```text
( o S. Köhler          )
  azure  forensik  ir
```

**Vorteil:** Informationsdichte ohne Interaktion. Such-Treffer sind im Chart direkt sichtbar.
**Nachteil:** Bei vielen Keywords pro Person wird Pool-Pill hoch. Keywords sind selten alle gleichzeitig relevant — sie sind Info, die der Suche dient, nicht dem Überblick.

#### Variante (b) — On Hover / On Tap als Tooltip

```text
( o S. Köhler ) <--- hover/tap
                     +-----------------------+
                     | Focus areas:          |
                     | azure-forensik        |
                     | incident-response     |
                     | malware-analysis      |
                     +-----------------------+
```

**Vorteil:** Pool-Pill bleibt schmal. Keywords on demand. Konsistent mit anderen Tooltip-Mustern.
**Nachteil:** Auf Mobile ist Hover nicht verfügbar — Tap müsste zwischen „Detail-Modal öffnen" und „Tooltip anzeigen" unterscheiden, was kollidiert.

#### Variante (c) — Nur in Such-Trefferliste sichtbar, nicht im Chart

```text
Chart-Pool-Pill: ( o S. Köhler )   <-- keine Keywords

Search-Tab Treffer:
  - S. Köhler   [ available ]
    azure-forensik, incident-response, malware-analysis
    Head of SOC pool
```

**Vorteil:** Chart bleibt minimal. Such-Tab ist sowieso der Kontext, in dem Keywords relevant werden.
**Nachteil:** Im Chart-View sieht man nicht, ob eine Person überhaupt Keywords hat — Such-Voraussetzung „weiß ich überhaupt, wonach ich suchen kann?" leidet.

#### **EMPFEHLUNG für Q7-ii:** Variante (b) — Tooltip on Hover, Tap-Detail auf Mobile

**Begründung:** Hält das Chart visuell ruhig (Keywords sind nicht Überblicks-Information). Auf Desktop löst Hover das Problem ohne Klick. Auf Mobile öffnet der bestehende Tap-→-Person-Detail-Pfad einen Modal, der die Keywords als Sektion anzeigt — kein zusätzlicher Interaktions-Pfad nötig. Variante (a) ist die Backup, falls Such-Tab später nicht das primäre Discovery-Pattern wird; aktuell genügt Suche + Detail-Modal.

## Strukturelle Lücken in der Spec (Bemerkung)

Beim Mockup-Schreiben sind zwei Punkte aufgefallen, die die Spec nicht explizit klärt — kein Blocker, aber vor Schritt 5 sollte das Patrick entscheiden:

1. **Pool ohne Lead-Slot:** Spec sagt „typisch ein Lead, ggf. mehrere". Was passiert, wenn ein Pool durch Rollen-Delete plötzlich keinen Lead mehr hat? Wegwerfen, an „verwaister" Sektion am Level-Ende rendern, oder Migration-Step der den Pool an einen Default-Lead bindet? Vorschlag: Verwaister Pool rendert in einer eigenen „Orphan pools"-Sektion außerhalb der Levels, mit Warn-Hinweis im Settings-Tab.

2. **Member-Reihenfolge im Pool:** Spec sagt „Keine Reihenfolge, Verfügbarkeit-getrieben". Heißt das, Render sortiert immer nach Verfügbarkeit (grün → gelb → grau → rot)? Oder alphabetisch mit Verfügbarkeit als sekundärer Sort? Letzteres ist für Stress-Lesbarkeit besser (bekannte Reihenfolge, Augen finden Person schnell). Erstes ist für Schnell-Triage besser („zeig mir die Grünen oben"). Vorschlag: Default alphabetisch, optionaler Toggle in Settings „sort by availability".

Beide Punkte sind klein genug, um in der Spec als Sub-Sektion 4.3 ergänzt zu werden.
